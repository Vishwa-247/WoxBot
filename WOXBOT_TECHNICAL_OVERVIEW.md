# WoxBot — Complete Technical Overview

> **Project type:** Agentic RAG (Retrieval-Augmented Generation) system  
> **Target users:** Students at Woxsen University  
> **Stack:** React 19 · FastAPI · LangGraph · FAISS · BM25 · MongoDB · Groq/OpenRouter

---

## 1. PROJECT OVERVIEW

WoxBot is an **AI-powered academic assistant** built for Woxsen University students. It lets students upload their course material (syllabi, lab manuals, lecture notes, question papers) as PDFs and then chat with that content using natural language.

**Problem it solves:**
Students waste time searching through long PDFs to find specific information. WoxBot lets them ask questions like *"What topics are in Unit 3 of my OS syllabus?"* or *"Calculate my CGPA"* or *"Explain the difference between TCP and UDP"* — and get instant, grounded, sourced answers.

**What makes it distinct from a generic chatbot:**
- Answers are grounded in **the student's own uploaded documents** — not hallucinated general knowledge
- A **10-node LangGraph agent pipeline** routes each query to the right tool (RAG, web search, calculator, study planner, etc.)
- Every answer is validated for hallucination before delivery
- The system streams tokens in real time via **Server-Sent Events (SSE)**

## 2. USER FLOW

```
Student uploads PDF
       │
       ▼
POST /api/ingest
  → PDF extracted (PyMuPDF)
  → Section-aware chunked (400 tokens/chunk)
  → Embedded (OpenRouter text-embedding-3-small, 1536-dim)
  → Saved to FAISS index + BM25 index (disk)
  → Chunk metadata saved to MongoDB
  → Auto-summary generated by LLM and stored
       │
       ▼
Student types a question in the chat UI
       │
       ▼
POST /api/chat (SSE stream)
  → Query rewritten using conversation history
  → Routed: document_qa | web_search | calculator | summarize | study_planner | unclear
  → [For document_qa / summarize]
       → FAISS semantic search (top-20)
       + BM25 keyword search (top-20)
       → RRF fusion → top-20 candidates
       → CrossEncoder reranker → top-8 chunks
       → LLM generates answer (streaming tokens via SSE)
       → Source mapping (post-hoc, embedding-based)
       → Hallucination validation (token overlap → embedding similarity → LLM)
  → Follow-up questions generated
  → Answer + sources + follow-ups streamed to frontend
       │
       ▼
React frontend renders markdown, source circles, follow-up buttons
```

## 8. DATA FLOW

### PDF Ingestion Flow

```
UploadFile (multipart)
       │
       ▼
save to data/raw/{filename}.pdf
       │
       ▼
PyMuPDF → extract text page-by-page → PDFDocument (pages, scanned_flags)
       │
       ▼
Section-aware chunker:
  detect headings (ALL CAPS / Title Case / numbered) → split into Sections
  → split each Section by 400-token limit with 80-token overlap
  → each Chunk: { chunk_id, text, filename, page, section_title, token_count }
       │
       ▼
Embedder (OpenRouter text-embedding-3-small):
  → batch embed all chunk texts → np.ndarray (n × 1536)
       │
       ├──→ FAISS IndexFlatIP: L2-normalize → add vectors
       │    SHA-256 hash tracks which files are indexed
       │    Persisted to vector_db/faiss.index + metadata.json
       │
       ├──→ BM25Okapi: tokenize → build index
       │    Persisted to vector_db/bm25.pkl
       │
       └──→ MongoDB:
            documents collection: { _id: sha256_hash, filename, pages, chunk_count, uploaded_at, summary }
            chunks collection: { doc_id, chunk_id, filename, page, section_title, text, token_count }
       │
       ▼
Auto-summary: LLM(first 10 chunks) → "What this doc contains + 3 questions"
  → saved to documents.summary field via update_document_summary()
       │
       ▼
Return IngestResponse: { filename, chunks_added, total_pages, summary }
Frontend InputBar shows toast with filename + chunk count + AI summary
```

### Chat Query Flow

```
User types query → InputBar.sendMessage()
       │
       ▼
useStream.startStream() — POST /api/chat
  body: { query, session_id, provider, model, selected_doc_ids }
  header: X-API-Key
       │
       ▼
chat.py: _generate_stream()

  Step 1: Query Rewriting
    conversation history (MongoDB/in-memory) + REWRITER_PROMPT
    → LLM rewrites query to be standalone (resolves pronouns, adds context)

  Step 2: Two-Tier Routing
    keyword_pre_route(query):
      → calc keywords? → "calculation"
      → study/exam keywords? → "study_planner"
      → summarize keywords? → "summarize"
      → doc keywords? → "document_qa"
      → web keywords? → "web_search"
      → ambiguous → LLM router with ROUTER_PROMPT → one of 6 routes

  Step 3: Resolve selected_doc_ids to filenames (MongoDB lookup)

  Step 4: Route Handling:

    [document_qa / summarize]
      hybrid_retrieve(query, selected_filenames)
        → embed query (OpenRouter)
        → FAISS search_filtered() OR bm25.search_filtered() if filenames given
          (over-retrieval × 5 → post-filter for FAISS; iterate for BM25)
        → RRF fusion → top-20 candidates
      rerank(query, candidates) → CrossEncoder → top-8 chunks
      LLM stream(RAG_SYSTEM_MSG + RAG_USER_MSG with context + history)
        → yields tokens → frontend renders in real time

    [calculation]
      safe_calculate(query) → regex-based arithmetic parser (no eval())

    [web_search]
      DuckDuckGo DDGS().text(query) → format results
      → LLM synthesizes answer from search snippets

    [unclear / clarify]
      list_documents() → build doc_list string
      → CLARIFY_PROMPT.format(query, doc_list) → LLM asks ONE clarifying question

    [study_planner]
      list_documents() → fetch all doc summaries from MongoDB
      → STUDY_PLAN_PROMPT → LLM builds structured study plan

  Step 5: Post-processing (for document_qa / summarize only)
    map_sources(answer, chunks, return_embeddings=True)
      → split answer into sentences
      → embed sentences + chunks (one API call, embeddings reused)
      → cosine similarity matrix → assign best-matching chunk per sentence
      → deduplicate → return source list + chunk_embs

    validate(answer, chunks, chunk_embs=chunk_embs)   ← reuses embeddings
      → token_overlap > 0.60 → skip → "high confidence"
      → token_overlap 0.15–0.60 → embedding similarity (precomputed)
        → sim ≥ 0.65 → "high confidence"
        → sim < 0.40 → "low confidence" (append disclaimer)
        → 0.40–0.65 → call LLM validator (expensive, only borderline cases)

  Step 6: Follow-up generation
    generate_followups(query, answer) → LLM returns JSON array of 3 questions
    yield "[FOLLOWUPS_START]" + JSON

  Step 7: Save to conversation memory

  Step 8: yield "[DONE]"

       │
       ▼
Frontend (useStream.js):
  reads SSE stream token by token
  on "[SOURCES_START]" → next event is JSON with sources
  on "[FOLLOWUPS_START]" → next event is JSON with follow-up questions
  on "[DONE]" → finalize message, attach sources + followups

       │
       ▼
MessageBubble renders:
  react-markdown (headings, lists, code blocks, bold, tables)
  SourceCircles (numbered badges, hover for page/section)
  Follow-up pill buttons (click → pre-fill InputBar)
```

## 11. AI / LLM INTEGRATION

### Embedding

- **Model:** `text-embedding-3-small` (OpenRouter, 1536-dimensional)
- **Used for:** chunked text embedding during ingestion, query embedding during retrieval, sentence embedding in source mapping & validation
- **Normalization:** L2-normalized before FAISS IndexFlatIP (so inner-product = cosine similarity)

### Retrieval (Hybrid)

Two complementary searches merged via **Reciprocal Rank Fusion (RRF k=60)**:

| Method | What it finds well | Implementation |
|---|---|---|
| FAISS semantic | Conceptually similar passages even if no keyword overlap | `IndexFlatIP.search()` with L2-normalized vectors |
| BM25 keyword | Exact terminology matches, acronyms, named entities | `BM25Okapi.get_scores()` |

Multi-doc filtering uses **over-retrieval (5× top-k) + post-filter** for FAISS (since IndexFlatIP has no native ID filter) and iterate-until-k for BM25.

### Reranking

- **Model:** `cross-encoder/ms-marco-MiniLM-L-6-v2` (sentence-transformers, ~22M params)
- Takes top-20 RRF candidates → scores each (query, chunk) pair jointly → returns top-8
- Loaded once at startup and cached in module-level variable to avoid repeated disk reads

### LLM (Generation)

**Unified adapter** in `llm.py` supports:

| Provider | Default Model | Interface |
|---|---|---|
| Groq | `llama-3.1-8b-instant` | OpenAI-compatible SDK |
| OpenRouter | `google/gemini-2.0-flash-001` | OpenAI-compatible SDK |
| Gemini | `gemini-1.5-flash` | google-genai SDK |
| Local (Ollama) | `phi3` | OpenAI-compatible SDK |

Both **sync** (`generate()`) and **async streaming** (`stream()`) are supported. The frontend model selector dropdown overrides the default per request.

### Agent Pipeline (LangGraph)

```
START
  │
  ▼
rewriter_node — Rewrites query to standalone form using conversation history
  │
  ▼
router_node — 2-tier: keyword rules → LLM router if ambiguous
  │
  ├─ document_qa → rag_node (FAISS+BM25+rerank+LLM)
  ├─ web_search  → search_node (DuckDuckGo + LLM synthesis)
  ├─ calculation → calc_node (safe arithmetic, no eval())
  ├─ summarize   → summarizer_node (retrieve 30 chunks, summarize)
  ├─ unclear     → clarify_node (ask one clarifying question with doc context)
  └─ study_planner → study_planner_node (read all doc summaries → structured plan)
           │
           ▼
validator_node — 3-tier hallucination check (only for RAG routes)
  │
  ▼
memory_node — Save (query, answer) to conversation buffer
  │
  ▼
END
```

### Hallucination Validation (3-Tier)

The validator runs **cheapest first** to avoid unnecessary API calls:

| Tier | Method | Condition | Cost |
|---|---|---|---|
| 1 | Token overlap ratio | `overlap > 0.60` → high confidence skip; `< 0.15` → fail | Free (regex) |
| 2 | Embedding cosine similarity | `sim ≥ 0.65` → high; `< 0.40` → low | 1 embedding call (chunk_embs reused from map_sources) |
| 3 | LLM judge | 0.40–0.65 borderline → call VALIDATOR_PROMPT | 1 LLM call (rare) |

## 12. PERFORMANCE AND DESIGN DECISIONS

### Decision 1: SSE over WebSockets
**Why:** SSE is simpler (HTTP/1.1, unidirectional, no upgrade handshake), sufficient for token streaming, and works through standard CDN/proxy setups. The bot never needs to receive data mid-stream.

### Decision 2: Hybrid Retrieval over pure vector search
**Why:** Pure semantic search misses exact keyword matches (e.g., "RR-interval", course codes, professor names). Pure BM25 misses paraphrased queries. RRF combination empirically outperforms either alone by 10–20% MRR on academic text.

### Decision 3: 2-Tier Router (keyword first, LLM second)
**Why:** LLM routing adds 200–500ms latency. ~90% of queries match keywords deterministically. The LLM router is only called when keywords are ambiguous, saving cost and latency.

### Decision 4: Precomputed chunk embeddings threading
**Why:** `map_sources()` already embeds the top-8 chunk texts to do sentence→chunk matching. The validator's `_embedding_similarity()` would naively re-embed the same 8 chunks. We pass `chunk_embs` through to save one OpenRouter API call (≈300ms) per answer.

### Decision 5: `hmac.compare_digest()` for API key check
**Why:** Prevents timing-attack side-channel leaks. Using `==` for secret comparison leaks information about how many characters matched.

### Decision 6: SHA-256 dedup for document ingestion
**Why:** Computing a SHA-256 hash of the PDF binary prevents re-ingesting the same file uploaded twice, avoiding duplicate vectors in FAISS and duplicate records in MongoDB.

### Decision 7: Section-aware chunking over RecursiveTextSplitter
**Why:** Academic PDFs have explicit structure (chapters, units, sections). Cutting blindly by character count would split mid-concept. Heading-aware chunking keeps each unit of knowledge together and prepends the section title to every chunk, giving the LLM structural context.

### Decision 8: CrossEncoder over colbert/bi-encoder for reranking
**Why:** CrossEncoder jointly encodes (query, passage) pairs — better relevance signal for short academic queries. The `ms-marco-MiniLM-L-6-v2` model is small (22M params) but competitive. Loaded once at startup (4–5s) then cached.

### Decision 9: MongoDB `$set` for summary update
**Why:** `save_document()` with a partial dict would overwrite `uploaded_at`. A dedicated `update_document_summary()` using `$set {summary: ...}` surgically updates only one field without touching other metadata.

## 13. POTENTIAL IMPROVEMENTS

| Area | Issue | Suggested Improvement |
|---|---|---|
| **FAISS filtering** | Over-retrieval (5× k) for multi-doc filtering is wasteful with many documents | Migrate to Qdrant or Weaviate which support native payload filtering |
| **Embeddings** | Every query embeds synchronously — blocks the async loop | Run `embed_query` in a thread pool via `asyncio.run_in_executor` |
| **BM25 persistence** | Entire BM25 index loaded from pickle on every query | Keep BM25 in memory as a module-level singleton (already done in `bm25_store.py`) |
| **Follow-up generation** | Separate LLM call after every answer adds ~500ms | Batch follow-ups into the same LLM call as the answer or pre-generate async |
| **Vector dedup** | SHA-256 is computed on entire file at ingest time | Store hash in MongoDB to avoid re-computing on server restarts |
| **No pagination on /sources** | `list_documents()` returns all docs with no limit | Add cursor-based pagination for large document libraries |
| **Frontend auth** | API key is hardcoded in the frontend `.env` | Implement session-based auth with JWT tokens |
| **OCR for scanned PDFs** | Scanned pages (< 50 chars/page) are flagged but not processed | Integrate Tesseract OCR or Document AI as fallback |
| **RAGAS evaluation** | Evaluation pipeline in `app/evaluation/` exists but is not automated | Hook into CI/CD to run RAGAS metrics on a golden test set on every deploy |
| **State in AgentState TypedDict** | `available_docs` and `doc_summaries` are never set in AgentState | Populate these in the async SSE handler and pass via state for the sync graph path |
| **Chunking quality** | Token estimation is char/4 approximation, not tiktoken | Use `tiktoken` for accurate token counts matching the embedding model |

## 14. INTERVIEW EXPLANATION (2 minutes)

> "WoxBot is an **agentic RAG system** I built for Woxsen University students. The core idea is simple: students upload their PDFs — syllabi, lab manuals, lecture notes — and they get a chatbot that can answer questions *grounded in their own documents*.
>
> On the **backend**, I use **FastAPI** with **Server-Sent Events** for real-time token streaming. The AI pipeline is built with **LangGraph** — a 10-node DAG where each query goes through: query rewriting, intelligent routing, specialized processing, hallucination validation, and memory saving.
>
> The **retrieval** is hybrid — I combine **FAISS semantic search** with **BM25 keyword search**, merge them using **Reciprocal Rank Fusion**, then rerank the top-20 results with a **CrossEncoder** to get the best 8 chunks. This hybrid approach handles both conceptual similarity *and* exact terminology matching, which matters a lot for academic content.
>
> I built a **2-tier router** — keyword rules first handle 90% of queries instantly, and only ambiguous queries hit the LLM router. This cuts routing latency from ~500ms to near-zero for most requests. The router dispatches to 6 specialized nodes: document Q&A, web search, calculator, summarizer, study planner, and clarifier.
>
> For **hallucination prevention**, I have a 3-tier validator — token overlap first (free), then embedding cosine similarity (reusing precomputed embeddings), then an LLM judge only for borderline cases. This means most answers are validated without a single extra API call.
>
> **MongoDB** stores document records, chunk metadata, and conversation history. **FAISS and BM25** are persisted to disk. The system supports **multi-document filtering** — students can pin specific docs and only retrieve from those.
>
> On the **frontend** — React 19, Vite, Tailwind CSS. I built a custom SSE reader that handles three special markers in the stream: `[SOURCES_START]` triggers source badge rendering, `[FOLLOWUPS_START]` renders click-to-send suggestion buttons, and `[DONE]` finalizes the message. Answers render as full Markdown with code blocks, tables, and lists.
>
> Key design decisions: SSE over WebSockets (simpler for one-directional streaming), section-aware chunking instead of RecursiveTextSplitter (preserves academic structure), and threading precomputed embeddings between source mapping and validation to eliminate a redundant API call.
>
> The whole system is dockerized, deployable to Render.com, and has an MCP server stub for future IDE tool-use integration."
